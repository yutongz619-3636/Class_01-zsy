# Flask 登录系统漏洞检测与修复报告

## 一、检测对象

- 框架：Python Flask
- 功能：首页、登录、登出、用户信息展示
- 检测重点：密码暴力破解
- 检测方法：人工代码审计与 Flask 测试客户端验证

## 二、原程序存在的问题

| 编号 | 问题 | 风险等级 | 原因与影响 | 修复措施 |
|---|---|---|---|---|
| 1 | 登录接口没有失败次数限制 | 高 | 攻击者可以自动提交大量密码组合，持续猜测用户密码 | 按“IP＋用户名”计数，失败 5 次后锁定 60 秒 |
| 2 | 密码明文存储并用 `==` 比较 | 高 | 源码泄露时所有用户密码会直接暴露 | USERS 只保存 scrypt 哈希，使用 `check_password_hash` 验证 |
| 3 | 首页显示用户密码 | 高 | 登录用户或旁观者可以直接看到敏感信息 | 模板删除密码项，只传递允许展示的字段 |
| 4 | 登录页 HTML 注释泄露管理员密码 | 高 | 查看网页源代码即可获得默认管理员账号 | 删除调试注释和所有默认密码提示 |
| 5 | Session 密钥硬编码且容易猜测 | 中 | 攻击者得到密钥后可能伪造 Flask Session | 优先读取 `SECRET_KEY` 环境变量，未配置时随机生成 |
| 6 | 默认开启 Flask Debug | 中 | 调试信息可能泄露代码和运行环境，某些情况下风险更严重 | 默认关闭，只在 `FLASK_DEBUG=1` 时开启 |
| 7 | CSS 注释格式错误 | 低 | `/ 注释 /` 不是合法 CSS 注释，可能导致样式解析异常 | 全部改为 `/* 注释 */` |

## 三、防暴力破解设计

程序使用 `LOGIN_ATTEMPTS` 字典保存登录状态，键为 `(客户端 IP, 用户名)`，值包含失败次数和锁定结束时间。

登录流程如下：

1. 用户提交用户名和密码。
2. 程序先检查该“IP＋用户名”是否处于锁定状态。
3. 若已锁定，返回 HTTP 429，并提示剩余等待时间。
4. 若未锁定，使用密码哈希验证登录信息。
5. 登录成功后清除失败记录并创建 Session。
6. 登录失败时次数加一；第 5 次失败后锁定 60 秒。

选择 60 秒是为了方便课堂演示。真实系统通常会使用 10 至 30 分钟锁定时间，并配合验证码、日志告警和分布式限流。

## 五、方案局限与后续改进

本次实现贴合基础课程，逻辑简单直观，但仍有以下局限：

- 失败记录只保存在内存中，重启服务后会丢失。
- 多进程或多服务器部署时，各进程之间不能共享计数。
- 分布式攻击者可以更换 IP；共享出口的多个正常用户也可能互相影响。
- 项目没有实现验证码、登录日志告警、CSRF 防护和 HTTPS。

正式系统建议使用 Redis 保存失败记录，增加基于账号和 IP 的双重限流，部署 HTTPS，并使用 Gunicorn 或 uWSGI 运行 Flask 应用。

## 六、结论

原程序存在明显的密码暴力破解与敏感信息泄露风险。修复后，连续失败登录会触发临时锁定，密码改为哈希验证，页面不再显示或注释泄露密码，Session 和 Debug 配置也得到加强。该版本适合作为“暴力破解漏洞检测与修复”的课程作业演示。
